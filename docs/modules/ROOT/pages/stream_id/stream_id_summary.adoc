The purpose of this demonstration is to show how Stream Identification (part of both IEEE 802.1CB and 802.1Qci) is used to map incoming frames to streams.  Scenarios as part of those other standards will be used to demonstrate how the switch can act on those streams.

Stream Identification operates by defining a series of streams.  Each stream has an associated set of rules used for matching.  Incoming frames are evaluated against the streams in numeric order (e.g. stream 1 is evaluated before stream 2, even if stream 2 was defined first).  The frame is assigned to the first stream where all rules are matched.  It is also possible for frames to not match any stream, in which no stream processing functions enabled in the switch will affect that frame.

== Common Details

Unless explicitly defined otherwise, for every scenario, every traffic item produced by the end points should be produced at a unique rate, using different power of 2's (100 Kbps, 200 Kbps, 400 Kbps, 800 Kbps, etc).  The flow meters should also be set to unique rates, and always lower than the slowest traffic production rate.  This guarantees that it is always clear which traffic items are being filtered, and by which flow meter.

== Scenario Details

=== Scenario 1 - Order-of-execution

*Purpose:* Demonstrate the order in which streams are checked for a match to each frame.  Per the standard, the streams should be checked in numerical order by ID, and the frame assigned to the first stream it matches, with no further streams checked for that frame.

*It is critical, for this scenario, to perform these steps in the order listed, even though it would be quicker to rearrange them*

. Create 1st stream
.. Create stream, with ID 10, that matches everything on ports 1,2
.. Create flow meter ID 1, CIR = 100 Kbps, CBS = 8192 bytes
.. Create stream filter ID 1 connecting stream ID 10 to flow meter ID 1
. Create 2nd stream
.. Create stream, with ID 5, that matches everything on ports 1,2
.. Create flow meter ID 2, CIR = 200 Kbps, CBS = 8192 bytes
.. Create stream filter ID 2 connecting stream ID 5 to flow meter ID 2
. Create 3rd stream
.. Create stream, with ID 20, that matches everything on ports 1,2
.. Create flow meter ID 4, CIR = 400 Kbps, CBS = 8192 bytes
.. Create stream filter ID 4 connecting stream ID 20 to flow meter ID 4
. Create traffic
.. Create traffic item from EP~1~, 800 Kbps, destination EP~3~
. Execute test
.. Observe which flow meter takes effect.  Is it lowest number, highest number, or most recently created?
... Should see traffic item receive rate 200 Kbps (stream ID 5, lowest number)

=== Scenario 2 - DMAC matching

*Purpose:* Demonstrate how frames can be matched to a stream based on the specified destination MAC, including with a mask for wild card matching.

. Define a stream based on DMAC using a mask to have some wild cards
.. DMAC = 00:16:01:00:00:01, Mask = 00:FF:00:00:00:FF
.. Ports 1 & 2
. Create 4 total traffic items, 2 each on EP~1~ and EP~2~
.. 1 traffic item from each EP~1~ and EP~2~, using separate DMACs, but each matching the DMAC+mask defined for the stream
... Traffic Item 1: EP~1~: 100 Kbps, DMAC 00:16:02:00:00:01
... Traffic Item 2: EP~2~: 200 Kbps, DMAC 01:16:00:FF:FF:01
.. 1 traffic item from each EP~1~ and EP~2~, using separate DMACs, that do not match
... Traffic Item 3: EP~1~: 400 Kbps, DMAC 00:16:01:00:00:02
... Traffic Item 4: EP~2~: 800 Kbps, DMAC 00:17:01:00:00:01
.. Given all of this traffic needs to get through, need to create 4 receiving devices on EP~3~ with those 4 DMAC addresses specified above, and make sure the switch is configured to know what port those are on.
. Create a flow meter, limited to 50 Kbps.
. Create a stream filter, that maps the flow meter to the stream.
. Execute test, should see:
.. Traffic Item 1+2 received rate = 50 Kbps
.. Traffic Item 3 received rate = 400 Kbps
.. Traffic Item 4 received rate = 800 Kbps

=== Scenario 3 - SMAC matching

*Purpose:* Demonstrate how frames can be matched to a stream based on the specified source MAC, including with a mask for wild card matching.

. Define a stream based on SMAC using a mask to have some wild cards, ports 1,2
.. SMAC = 00:12:01:00:00:01, Mask = 00:FF:00:00:00:FF
. Create 4 total traffic items, 2 each on EP~1~ and EP~2~, all to the same destination MAC on EP~3~
.. 1 traffic item from each EP~1~ and EP~2~, using separate SMACs, each matching the masked SMAC
... Traffic Item 1: EP~1~: 1000 Kbps, SMAC 00:12:02:00:00:01
... Traffic Item 2: EP~2~: 2000 Kbps, SMAC 01:12:00:FF:FF:01
.. 1 traffic item from each EP~1~ and EP~2~, using separate SMACs, that do not match the masked SMAC
... Traffic Item 3: EP~1~: 4000 Kbps, SMAC 00:12:01:00:00:02
... Traffic Item 4: EP~2~: 8000 Kbps, SMAC 00:13:01:00:00:01
. Create a flow meter, limited to 500 Kbps.
. Create a stream filter, that maps the flow meter to the stream.
. Execute test, should see:
.. Traffic Item 1+2 received rate = 500 Kbps
.. Traffic Item 3 received rate = 4000 Kbps
.. Traffic Item 4 received rate = 8000 Kbps

=== Scenario 4 - IP based matching

*Purpose:* Demonstrate how IP frames can be matched to a stream based on the specified source IP address, destination IP address, and port.  This can be useful to differentiate by application, where different applications use different ports.  This all demonstrates making a "catch all" stream as the last stream, so that every frame matches to something.  This would for example be a way to get a switch to simply drop all unexpected traffic.

. Setup network stack such that EP~1~ and EP~2~ can each send and receive UDP/IPv4 traffic, and have IP addresses IP1 and IP2 respectively
. Create 6 streams, per table below, each mapped to an associated flow meter via a stream filter, setup to drop any frames above the specified steady-state rate.  The table also indicates if we expect any frames to match to this stream when the test is executed, based on the traffic item definitions specified further down.  
.. Use CBS of 2048 bytes for all flow meters
+
.IP Based Matching Stream Rules
|===  
|Stream ID |Source IP |Destination IP |Destination Port |Flow Meter CIR |Switch Port(s) |Matching SDUs?
|1 |Any |Any |3300 |10 Kbps |1,2 |Yes
|2 |IP1 |Any |Any |20 Kbps |1,2 |Yes
|3 |Any |IP2 |1000 |30 Kbps |1,2 |No
|4 |Any |IP2 |4000 to 4400 |40 Kbps |1,2 |Yes
|5 |IP2 |IP1 |3000 |50 Kbps |1,2 |Yes
|6 |Any |Any |Any |0 Kbps |1,2 |Yes
|===
+
. Create 10 sets of traffic items, with an indication of which stream it is expected to match to
+
.IP Based Matching Traffic Items
|===  
|Traffic Item |Source IP |Destination IP |Destination Port |Rate |Expected Stream
|1 |IP1 |IP1 |1000 |100 Kbps | 2 
|2 |IP1 |IP1 |1100 |200 Kbps | 2 
|3 |IP1 |IP2 |2000 |400 Kbps | 2 
|4 |IP1 |IP2 |2200 |800 Kbps | 2 
|5 |IP2 |IP1 |3000 |1600 Kbps | 5
|6 |IP2 |IP1 |3300 |3200 Kbps | 1
|7 |IP2 |IP2 |4000 |6400 Kbps | 4
|8 |IP2 |IP2 |4400 |12800 Kbps | 4
|9 |IP2 |IP2 |5000 |25600 Kbps | 6
|10 |IP2 |IP1 |5000 |51200 Kbps | 6
|===
+
. Run the scenario.
.. Verify in switch statistics that SDUs are matching to the expected streams
.. Verify via destination bit-rate statistics, that the frames are matching and being filtered as expected
... Traffic items 1, 2, 3, 4 should total 20 Kbps
... Traffic item 5 should be 50 Kbps
... Traffic item 6 should be 10 Kbps
... Traffic item 7, 8 should total 40 Kbps
... Traffic item 9, 10 should drop 100% of frames
.. Check switch resource availability vs the base switch config, to see what % of capabilities this is using.

=== Scenario 5 - Source Port

*Purpose:* Demonstrate the switch can assign frames to a stream based on which physical port the frame was received.  Supporting this might not be required by the standard, although there is a note that implies some way to do this (see https://hal.science/hal-04159172v4 section 2.1).  Note that also complete fault tolerance cannot be delivered without this capability - there must be some way to prevent a device from spoofing valid frames from a different device.

. Create stream ID 1 that matches anything, on any port EXCEPT ports 1,2,21
. Create stream ID 2 that matches anything on port 1
. Create stream ID 3 that matches anything on port 2
. Create separate flow meters for each stream ID, and map via stream filters
.. CIR = 0 Kbps, 20 Kbps and 80 Kbps, respectively
.. CBS = 2048 bytes for each
. Create 4 total traffic items, 2 each on EP~1~ and EP~2~.  The traffic on each EP should be identical.
.. Traffic Item 1: EP~1~: 100 Kbps, SMAC 00:12:01:00:00:01, DMAC 00:16:01:00:00:01, VLAN 0 priority 1
.. Traffic Item 2: EP~1~: 200 Kbps, SMAC 00:13:01:00:00:01, DMAC 00:17:01:00:00:01, VLAN 0 priority 2
.. Traffic Item 3: EP~2~: 400 Kbps, SMAC 00:12:01:00:00:01, DMAC 00:16:01:00:00:01, VLAN 0 priority 1
.. Traffic Item 4: EP~2~: 800 Kbps, SMAC 00:13:01:00:00:01, DMAC 00:17:01:00:00:01, VLAN 0 priority 2
.. Given all of this traffic needs to get through, need to create 4 receiving devices on EP~3~ with those 4 DMAC addresses specified above, and make sure the switch is configured to know what port those are on.
. Execute test, should see:
.. Traffic Item 1+2 received rate = 20 Kbps
.. Traffic Item 3+4 received rate = 80 Kbps


=== ON HOLD: Additional Scenarios

It is not clear if these capabilities would ever be used, and therefore there are currently no plans to demonstrate them:

* Multicast / Broadcast / Unicast filters
* Inner VLAN tags, VLAN ID, tag type, DEI
* Outer VLAN tag (but do have scenarios defined)
** Setup stream S1 with VLAN tag not allowed, on both ports 1 & 2
** Create flow meter at rate 0 (drop everything), attached to stream S1
** Setup stream S2 with VLAN tag required, any tag type, VLAN 2, VLAN mask 0xFFF, PCP 0, PCP mask 0, DEI any, on both ports
** Create flow meter at rate X1, attached to stream S2
** Setup stream S3 with VLAN tag required, any tag type, VLAN 0, VLAN mask 0, PCP 3, PCP mask 0x7, DEI any, on port 1 only
** Create flow meter at rate X2, attached to stream S3
** Setup stream S4, that matches everything, ports 1 & 2.  
** Create flow meter at rate X3, attached to stream S4
** Create 7 traffic items per end point.  One with no VLAN tag.  One with VLAN 2 priority 1.  One with VLAN 2 priority 3.  One with VLAN 0 priority 1.  One with VLAN 0 priority 3.  One with VLAN 3 priority 3.  One with VLAN 3 priority 1.
** See what happens!   In particular want to see if PSFP treats VLAN 0 as "no VLAN" or "VLAN 0".  Also want to see if this filtering happens after VLAN 2 is assigned.
* Protocol filters for EtherType, LLC, SNAP, and IPv6
* IPv4 filter settings for Prefix Length, DSCP, Fragment, and Protocol
