This document provides specific instructions for implementing the TAS demo with the Keysight Novus One Plus network emulator acting as the end-points, and the Microchip VSC5640EV switch hardware running iStax software.

The specific port assignments shown may need to be modified to match how the Keysight is cabled to the switch.  For these steps, it is assuming Keysight ports 5-8 are cabled to switch ports 1-4.

==== Test Execution & Results

Now that TAS gating is enabled on the switch, we expect to see an effect on latencies.

.Scenario 5 latencies
image::802.1Qbv/keysight_microchip_scenario5/scenario5_keysight_latency.png[]

The scheduled traffic latencies are reduced relative to scenario 1, which had the same traffic generation configuration but TAS gating disabled on the switch.  (Average latency reduced from 8 micro-seconds to 2 micro-seconds).  The gating reserves the time for these frames, so the switch never has to wait for another frame to complete first.  This can in particular be seen with the reduced average latency, which is very close to the minimum latency.  The max latency is a bit higher, and it is believed this is due to a single longer latency frame at the start of test while things are still being synchronized.  

As a sanity check, set the 4 scheduled traffic streams on the keysight to all output at the same time, and keep TAS gating enabled on the switch.  Latencies for the 4 scheduled traffic streams should be ~0, 250, 500, and 750 micro-seconds respectively.  Keysight is outputting all 4 frames at the start of each network cycle, and they get queued by the switch waiting for the gate to open.
