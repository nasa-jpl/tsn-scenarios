This document provides specific instructions for implementing the Stream Identification scenarios with the Keysight Novus One Plus network emulator acting as the end-points, and the Microchip VSC5640EV switch hardware running iStax software.

It is assumed that there is basic familiarity with the Keysight and switch.  If not, the TAS scenario instructions are much more detailed and provide specific steps for how to create topologies and traffic items in Keysight, etc.

All scenarios assume the base switch configuration has already been performed.  Instructions can be found here: xref:switch-config.adoc[].

All scenarios assume the Keysight is starting with the base 802.1AS time synchronization setup, found here: xref:802.1AS/index.adoc[].

The scenarios will just list out the steps of how to configure the switch, without detailed screenshots for each step. Note that the scenarios require both streams to be defined, as well as flow meters and stream filters.  Similar screen shots covering those are found in the PSFP scenario documentation.

== Detailed Steps, Stream Mapping
=== Scenario 1
Rule Order-Of-Execution

==== Detailed Switch Configuration

See screenshots earlier in this scenario, and in the PSFP venue-specific documentation, for how to configure the switch.  No further information should be needed beyond what is provided by the general scenario description.

==== Detailed Novus Configuration

* Create a single traffic flow
** EP~1~ transmitting to EP~3~
** 800 Kbps rate
** fixed 64 byte frames

==== Test Execution and Results

The expectation is that the traffic item will only see 200 Kbps get through, as the switch should be checking the frame against stream rules in ascending numeric order, regardless of the order in which they were created.

The results do match that as shown in the screenshots below.  

Note that the bit rate being received slightly exceeds 200 Kbps.  It is not fully understood why that is.  Ticket was opened https://github.jpl.nasa.gov/hpsc/tsn-scenarios/issues/3.  The same effect is expected for any tests using flow meters, but will not repeat this text for every scenario.  This idiosyncrasy will just be captured here.

=== Scenario 2
Destination MAC matching

==== Detailed Switch Configuration

General scenario description should be sufficient.  Need to add the 4 specific destination MAC addresses being used as static entries to the MAC table.  Recommend removing them at end of test.

==== Detailed Novus Configuration

* For the most part, the general scenario description should be sufficient.
* Need to have 4 destination MAC addresses on EP~3~.  To do this, you can add 4 additional Ethernet protocol stacks to that device group, and then set the MAC address of each.



==== Test Execution and Results

Results match expected - the 2 traffic flows matching the DMAC, combined, are reduced to ~50 Kbps.  The other traffic flows are not affected.

=== Scenario 3
Source MAC matching

==== Detailed Switch Configuration

General scenario description should be sufficient.

==== Detailed Novus Configuration

* For the most part, the general scenario description should be sufficient.
* Need to have 2 destination MAC addresses each on EP~1~ and EP~2~.  To do this, you can add additional Ethernet protocol stacks to that device group, and then set the MAC address of each.  To avoid issues, don't use the ethernet stack used for time sync.  Changing the MAC address of that might prevent time sync from working.
* Note that the source MAC address 01:12:00:ff:ff:01 did not work.  Issue https://github.jpl.nasa.gov/hpsc/tsn-scenarios/issues/4 created.  Used 00:12:00:ff:ff:01 instead which works and still demonstrates SMAC matching sufficiently.

==== Test Execution and Results

Results match expected - the 2 traffic flows matching the SMAC, combined, are reduced to ~500 Kbps.  The other traffic flows are not affected.
