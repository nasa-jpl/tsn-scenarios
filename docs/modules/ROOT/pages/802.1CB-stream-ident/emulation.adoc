This document provides specific instructions for implementing the Stream Identification scenarios with the Keysight Novus One Plus network emulator acting as the end-points, and the Microchip VSC5640EV switch hardware running iStax software.

It is assumed that there is basic familiarity with the Keysight and switch.  If not, the TAS scenario instructions are much more detailed and provide specific steps for how to create topologies and traffic items in Keysight, etc.

All scenarios assume the base switch configuration has already been performed.  Instructions can be found here: xref:switch-config.adoc[].

All scenarios assume the Keysight is starting with the base 802.1AS time synchronization setup, found here: xref:802.1AS/index.adoc[].

=== Scenario 4 - IP based matching

==== Detailed Switch Configuration

The only setup on the switch, on top of the base switch configuration, is to defined the streams & flow meters, and then stream filters that map them together.  2 screenshots below show examples of how to configure streams 1 & 2.  

The remaining streams should be straight-forward to configure.  How to create the flow meters and stream filters are covered by the previous scenarios.  The table in the general scenario description should be sufficient, with the following basic steps

* Create the 6 streams with matching based on the Source IP, Destination IP, and Destination Port columns from the general scenario description
** To match any IP address, use the default prefix length = 0 (this is the number of most-significant bits of the address that must match)
** To match a specific IP address, use prefix length = 32, meaning all bits must match
** To match a subnet, use prefix length = 24, meaning the most significant 24 bits must match.  In other words if IP address is entered in the stream ID dialog as x.y.z.0, frames received by the switch with any IP address x.y.z.* will match to that stream
** Don't forget to select the ports the frames will be received on.  In the example, I just always picked both ports 1 & 2 rather than the specific port it will be received on.
* Create 6 flow meters with the CIR value specified in the general scenario description (suggest using numbering matching the stream ID it will map to)
* Create 6 stream filters, mapping each stream to the associated flow meter

==== Detailed Novus Configuration

Start with a new session.  Time synchronization does not need to be configured for this to work.

* Create the topology
** Create the 3nd end points / topologies
** EP~1~ and EP~2~ should each have 1 IPv4 protocol stack
** EP~3~ should have 2 IPv4 protocol stacks
* While optional, it can be very useful to rename each topology, device group, port, and protocol stack so the numbering all aligns with the standard topology and scenario description.
** This is time consuming but well worth it when later creating traffic classes, and trying to debug issues!
** Sometimes you have to temporarily rename something and then rename it again.  You cannot have for example 2 topologies with the same name, and one topology may already have the name you want.  For example, Keysight may name the device groups, "Device Group 1", "Device Group 2", and "Device Group 3".  But with the numbering not aligned to the EP~n~ numbering.  So you have to reverse the names "Device Group 2" and "Device Group 3".  So you may need to rename "Device Group 2" to "Device Group 100", then rename "Device Group 3" to "Device Group 2", then rename "Device Group 100" to "Device Group 3".
** Not all items can be renamed through the GUI, some must be through the "API Browser"
** The following screenshots show basic steps to be able to rename everything.
** Note that when this scenario was constructed, something messed up the ports, and they had to be recreated.  It is possible renaming the ports did this.  In the final provided example, the ports were not renamed as a result.  Your mileage may vary.
