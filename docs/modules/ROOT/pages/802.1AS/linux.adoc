= Linux Venue

== Overview
* For the TSN switch, the base configuration covers basic time sync.  See xref::base_switch_config.adoc[]
* Both automated and manual configuration options are available to enable 802.1AS time synchronization on Linux systems, this guide walks through both of them.
* This guide is focused on Ubuntu Linux, but any platform that supports the `linuxptp` package may also be compatible, although with limited support and testing.

== Detailed steps to setup and enable time synchronization

=== Automated Ansible configuration
The easiest way to configure a set of Linux systems for 802.1AS time synchronization is to use the link:https://github.jpl.nasa.gov/hpsc/tsn-scenarios/tree/main/ansible#ansible[Ansible playbook] provided in the TSN scenarios repository. This playbook will install the necessary packages and configure the system for 802.1AS time synchronization. To do this you will need to designate:

* One device/hostname to serve as the grand master clock source
* One device/hostname to host the PTP process logging server
* One device/hostname to be the Ansible controller
* One or more devices to be configured as PTP clients or end instances

Devices on the network can serve as multiple roles at the same times, with the exception of the grand master clock which cannot simoultaneously be a PTP end instance. For this guide, we will have the Ansible controller, PTP logging server, and grand master clock all on the same device, and the PTP end instance on a separate device.

[NOTE] 
.Clock Source
====
This grand master doesn't necessarily have to have accurate GPS synchronized time for TSN features to work, however it is recommended if your intended use case requires accurate real world time.
====

.Step 1 Install Ansible and systemd on the controller device:
[source,sh]
----
sudo apt update
sudo apt install ansible systemd
----

.Step 2 Clone the TSN scenarios repository:
[source,sh]
----
git clone https://github.jpl.nasa.gov/hpsc/tsn-scenarios.git
cd tsn-scenarios/ansible
----

.Step 3 Modify the `inventory` file to include the devices you want to configure. 
[source,sh]
----
$EDITOR inventory
----
Copy and modify the link:https://github.jpl.nasa.gov/hpsc/tsn-scenarios/blob/main/ansible/inventory[ansible inventory file]:

[source,yml]
----
include::example$802.1AS/ansible-inventory.yml[]
----

Sections with `USER` comments indicate where you should add your device information, the comments explain what each section does. The inventory file is designed so that it can be used to configure as many devices as you want, with each device being an "end system" or 'es' for short. If your configuration needs more or less devices you can add entries at the bottom of the file for each end system, defined in the same way as the others in the file. Importantly, you also need to add them under `children:` of the `eis:` and `duts:` section of the inventory.

To match the example in this guide, make the following changes to the default inventory file:

* Change `logging_server: work-desktop` to `logging_server: hpsctsnes1` (or the specific hostname of your master clock / ansible controller / logging server).
* Change `ansible_user: ansible` to `ansible_user: your_username` note that this user must have passwordless sudo access on the devices you are configuring.
* Delete `es2:` under `children:` of the `duts:` and `eis:` section.
* Delete the entry for `es2:` and its associated host.
* Change all mentions of `hpsctsnesX` hostnames to match those of your devices.

.Step 4 Run the Ansible logging setup playbook:
[source,sh]
----
ansible-playbook logging.yml
----

.Step 5 Test the logging server:
From one of the end systems, run the following command:
[source,sh]
----
systemd-cat echo test
----
On the logging server, check the logs:
[source,sh]
----
sudo journalctl --directory /var/log/journal/remote --merge -n1
----
Expected output:
[source,sh]
----
user@logging-server:~$ sudo journalctl --directory /var/log/journal/remote --merge -n1
Jun 09 17:08:13 hpsctsnes1 echo[39637]: test
----
.Step 6 Monitor the logs on the logging server:
[source,sh]
----
sudo journalctl --file /var/log/journal/remote/all.journal -mf _COMM=ptp4l
----
.Step 7 Run the Ansible playbook to configure the devices for PTP:

ansible-playbook AS-start.yml

=== Manual configuration
Similar to the Ansible configuration, you will need to designate and grand master and an end instance.

.Step 1 Install the `linuxptp` package  on each device:
[source,sh]
----
sudo apt update
sudo apt install linuxptp
----

.Step 2 Ensure NTP is stopped for *end instances* (do not stop NTP on the grand master):
[source,sh]
----
sudo timedatectl set-ntp false
----

.Step 3 Download the ptp4l-gm.conf and ptp4l-ei.conf files from the TSN scenarios repository onto their respective devices:
[source,sh]
----
wget https://github.jpl.nasa.gov/raw/hpsc/tsn-scenarios/main/ansible/roles/AS/templates/ptp4l-gm.conf
----
[source,sh]
----
wget https://github.jpl.nasa.gov/raw/hpsc/tsn-scenarios/main/ansible/roles/AS/templates/ptp4l-ei.conf
----

You will need to replace the last line of these files to match the network interface you are using. For example, if your network interface is `enp1s0`, the last line should be:
[source,sh]
----
[enp1s0]
----

.Step 4 Check if any other PTP processes are running:
[source,sh]
----
pgrep ptp4l
pgrep phc2sys
----
If any processes are running, stop them:
[source,sh]
----
sudo kill [pid]
----

.Step 5 Start `phc2sys` on both devices (in a separate terminal):
[source,sh]
----
sudo phc2sys -s enp1s0 -c CLOCK_REALTIME --step_threshold=1 --transportSpecific=1 -w -m
----

.Step 6 Start `ptp4l` on the grand master device:
[source,sh]
----
ptp4l -i enp1s0 -f ptp4l-gm.conf -ml6
----

.Step 7 Start `ptp4l` on the end instance device:
[source,sh]
----
ptp4l -i enp1s0 -f ptp4l-ei.conf -ml6
----

[NOTE] 
.Network Interface
====
You may want to change `enp1s0` to the name of the network interface connected to the TSN switch. You can find the name of interfaces by running `ip a`
====

=== Additional verification, testing, and analysis using `tcpdump`
You can use `tcpdump` to verify that the PTP packets are being sent and received correctly. This can be useful for debugging and analysis.

.Step 1 Install `tcpdump`:
[source,sh]
----
sudo apt update
sudo apt install tcpdump
----
.Step 2 Start `tcpdump` on a device:
[source,sh]
----
sudo tcpdump -i enp1s0 -w log.pcap
----
[NOTE] 
.`tcpdump` options
====
There are various ways of using `tcpdump` to monitor network traffic, it's essentially a CLI version of Wireshark. The command above captures traffic and stores it to a .pcap (packet capture) file which we will open with a Wireshark client, but see the tcpdump link:https://www.tcpdump.org/manpages/tcpdump.1.html[man page] for more options.
====
.Step 3 Close `tcpdump` and analyze the log file:
Terminate `tcpdump` and analyze the log file. We will use Wireshark, but you can also use tshark or even `tcpdump` itself to analyze the log file.
